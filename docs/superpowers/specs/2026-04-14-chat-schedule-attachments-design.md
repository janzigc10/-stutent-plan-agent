# 聊天课表附件闭环设计文档

## 1. 目标

补齐聊天页中的课表附件发送闭环，让用户可以像常见聊天工具一样先选择附件、在输入区确认后再发送，并在发送后直接收到课表解析确认卡片。

本次设计只解决“聊天里的课表附件”这条链路，不扩展为通用文件消息系统。

## 2. 现状与问题

当前实现存在以下断点：

- 附件按钮选中文件后会立刻调用 `/api/schedule/upload`
- 上传成功后前端会直接拼接一条普通文本消息发到 WebSocket
- 输入区不存在“待发送附件”状态，用户无法确认、删除或分批发送
- 多张课表图片不会作为同一次导入处理
- 用户看到的结果更像“伪造一条文本提示”，而不是正式的聊天附件消息

后端已有能力并非空白：

- `POST /api/schedule/upload` 已支持图片与 Excel 上传
- 上传后会返回 `file_id`、`kind`、`courses`
- Agent 工具已有 `parse_schedule` 与 `parse_schedule_image`
- Agent 可通过 `ask_user` 返回确认卡片

因此本次实现的重点不是重做解析能力，而是补齐前端输入态、上传编排和最小必要的后端聚合逻辑。

## 3. 范围

### 3.1 本次包含

- 聊天输入区支持课表附件暂存
- 支持图片课表与 Excel/WPS 课表文件
- 一次消息最多 3 个附件
- 附件必须同类型发送，不允许图片与表格混发
- 多张图片作为同一次导入处理，最终生成一个确认卡片
- 发送成功后进入现有 Agent 解析链路

### 3.2 本次不包含

- 语音输入能力补齐
- 任意类型文件上传
- 聊天消息协议的大规模重构
- 视觉系统 / 组件库重构
- 聊天外其他页面的导入入口改造

## 4. 用户交互

### 4.1 选择附件

- 用户点击聊天页“附件”按钮
- 可选择图片文件或 Excel/WPS 文件
- 选择后先进入输入区上方的“待发送附件区”
- 这一阶段不上传、不触发解析、不发送 WebSocket 消息

### 4.2 待发送附件区

待发送附件区负责展示当前草稿中的附件：

- 显示文件名
- 显示附件数量
- 显示当前附件类型（图片或表格）
- 支持逐个删除
- 支持全部清空

约束：

- 图片最多 3 张
- Excel/WPS 一次只允许 1 个文件
- 若已选择图片，则只能继续加图片
- 若已选择表格，则不能再加图片或第二个表格

### 4.3 发送

- 用户点击“发送”后才开始真正上传
- 若当前只有附件、没有文本，也允许发送
- 前端先执行上传，再将上传结果组织成聊天消息送入现有 Agent 流程
- 发送成功后清空待发送附件区
- 发送失败时保留待发送附件区，允许用户重试或删除

### 4.4 发送后的聊天反馈

发送成功后，聊天流中应出现：

- 一条用户侧消息，表明已发送几张图片或一个课表文件
- Agent 继续走解析逻辑
- 最终返回确认卡片，供用户确认导入内容

对于用户来说，效果应为：

- “先挂附件”
- “点发送”
- “看到解析确认卡片”

## 5. 技术方案

### 5.1 方案选择

采用“保留现有上传接口 + 前端补待发送附件层 + 后端补多图聚合”的最小方案。

原因：

- 复用现有 `/api/schedule/upload`
- 复用现有 `parse_schedule` / `parse_schedule_image`
- 不改 WebSocket 握手与主协议
- 更适合先恢复功能闭环

### 5.2 前端状态模型

聊天 store 或聊天页本地状态新增轻量附件草稿结构：

```ts
type AttachmentKind = 'image' | 'spreadsheet'

interface PendingAttachment {
  id: string
  file: File
  name: string
  kind: AttachmentKind
  previewUrl?: string
}
```

需要支持的状态操作：

- 添加附件
- 删除单个附件
- 清空附件
- 校验是否允许继续添加
- 发送完成后重置

本次不引入独立全局上传模块，也不新增复杂的上传队列状态机。

### 5.3 上传编排

点击发送时按以下顺序执行：

1. 读取当前草稿中的附件集合
2. 校验数量与类型约束
3. 调用上传接口
4. 获取统一的 `file_id` 与 `kind`
5. 在前端聊天列表中追加一条用户消息
6. 通过 WebSocket 发送一条标准文本提示，引导 Agent 调用对应解析工具
7. 等待后端回传 `ask_user` 确认卡片

这轮仍沿用文本提示词触发 Agent，不新增二进制 WebSocket 传输。

### 5.4 上传接口

现有 `/api/schedule/upload` 需要扩展为支持同次多图导入：

- 请求允许多个 `file`
- 若为图片，允许 1 到 3 张
- 若为表格，仅允许 1 个
- 若混发，返回 400
- 若为多张图片，后端先逐张解析，再合并为同一个导入结果
- 合并结果写入同一个 schedule upload cache，返回一个 `file_id`

返回结构维持现有思路，增加对多文件的表达即可，例如：

```json
{
  "file_id": "uuid",
  "kind": "image",
  "count": 5,
  "source_file_count": 3,
  "courses": []
}
```

其中：

- `count` 表示识别出的课程数量
- `source_file_count` 表示本次上传的源文件数量

### 5.5 Agent 触发方式

前端上传成功后，仍发送一条标准消息给聊天后端：

- 表格：提示解析指定 `file_id`
- 图片：提示解析指定 `file_id`

提示词应尽量稳定，避免靠自然语言模糊猜测。推荐统一使用类似：

- `我上传了课表文件 file_id=<id>，请解析并展示确认卡片。`
- `我上传了课表图片 file_id=<id>，请解析并展示确认卡片。`

这样可以最大化复用现有 Agent 工具调用能力。

## 6. 组件与接口边界

### 6.1 前端

#### `frontend/src/pages/ChatPage.tsx`

负责：

- 文件选择
- 待发送附件区渲染
- 发送按钮行为编排
- 语音按钮继续保持现状，不纳入本次范围

#### `frontend/src/api/client.ts`

负责：

- 将现有单文件上传改为多文件上传
- 返回包含 `file_id`、`kind`、`count`、`source_file_count`

#### `frontend/src/stores/chatStore.ts`

负责：

- 继续维护消息、进度、ask_user、错误状态
- 若附件草稿放在 store 中，则只增加最小必要字段与操作
- 若附件草稿放在页面本地状态中，则 store 不必承担文件对象管理

推荐优先使用页面本地状态，避免把 `File` 对象写入全局 store。

### 6.2 后端

#### `app/routers/schedule_import.py`

负责：

- 接收多个上传文件
- 执行类型与数量校验
- 图片多文件聚合解析
- 写入统一缓存并返回结果

#### `app/services/schedule_upload_cache.py`

负责：

- 继续缓存按 `file_id` 聚合后的课表结果
- 无需感知源文件细节，只存最终聚合结果

#### `app/agent/tool_executor.py`

原则上不需要大的结构调整。

只要上传接口仍保证返回一个可消费的 `file_id`，`parse_schedule` / `parse_schedule_image` 现有逻辑即可继续使用。

## 7. 错误处理

### 7.1 前端本地校验

在真正上传前先拦截：

- 超过 3 个附件
- 图片与表格混发
- 表格超过 1 个
- 不支持的类型

这些错误应立即在聊天页可见，而不是等到服务器报错。

### 7.2 上传失败

若上传失败：

- 不清空待发送附件区
- 不追加“已发送”用户消息
- 显示明确错误文案
- 允许用户直接再次点击发送重试

### 7.3 解析失败

若上传成功但后续 Agent 未返回确认卡片：

- 沿用现有聊天错误链路
- 后续可配合聊天链路超时与错误事件增强
- 该问题不在本次设计主范围内，但前端代码应避免吞错

## 8. 测试策略

### 8.1 前端测试

需要覆盖：

- 选择图片后进入待发送附件区
- 选择表格后进入待发送附件区
- 混发被阻止
- 图片超过 3 张被阻止
- 删除单个附件
- 点击发送后先调用上传，再发送 WebSocket 消息
- 上传失败时附件仍保留

### 8.2 后端测试

需要覆盖：

- 单个表格上传成功
- 单张图片上传成功
- 多张图片上传成功并返回单个 `file_id`
- 混合图片和表格上传返回 400
- 图片超过 3 张返回 400
- 表格超过 1 个返回 400

## 9. 验收标准

满足以下条件即可视为本次闭环完成：

1. 聊天页可先挂载附件，再由用户主动发送
2. 一次最多支持 3 张课表图片，或 1 个 Excel/WPS 文件
3. 不允许图片与表格混发
4. 多张图片作为同一次导入处理
5. 发送后用户能在聊天中收到确认卡片
6. 失败时用户可重试，待发送附件不会无故丢失

## 10. 后续工作

本次闭环完成后，可继续拆分后续任务：

- 语音输入真正接入聊天发送链路
- 聊天链路超时与显式错误事件
- 日历页交互和接口联调
- UI 系统与视觉重构
